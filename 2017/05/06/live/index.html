直播相关协议对比

国内直播常用协议分析对比。

接触直播领域也有一段时间了，这次来总结一下和直播相关的一些通信协议，对它们做一个比较。就国内而言，比较常用的协议包括RTMP（Real Time Messaging Protocol），HTTP-FLV，HLS（Http Live Streaming）。

RTMP

这个协议是一个应用层协议，建立在TCP层之上，协议内容非常多，流程也比较复杂，光是应用层的握手就有点绕了。国内直播网站，如斗鱼，主播上行基本都是使用OBS进行推流，走的就是RTMP协议。服务端下行也有现成的开源软件，如nginx-rtmp来传输FLV格式的视频流，不过用起来还是有坑的，这点需要注意。总体来说，RTMP协议还是以上行推流使用居多，下行一般使用HTTP传输FLV或HLS。

优点：

  1. 延延时。
  2. RTMP使用场景更加广泛，包括视频会议等，它包含了控制播放器行为的一些命令。

缺点：

  1. 1935端口，视频流对于某些防火墙可能无法通过。
  2. 协议复杂。

HTTP-FLV

HTTP-FLV一般用来传输FLV格式的视频流，这是应用最为广泛的一种协议。

优点：

  1. 基于HTTP，在CDN中的支持较好，开发、调试成本要小一些。
  2. 低延时，延迟与RTMP相当。
  3. 80端口，不会被防火墙阻拦。
  4. 302重定向，方便CDN配合调度系统的使用。

HLS

苹果公司搞的一套基于HTTP的协议。不同于RTMP和HTTP-FLV，HLS主要是传输一个个小的ts视频文件，每个ts文件都可以单独播放。除了ts文件，还有一种m3u8索引文件，是对ts文件的索引，格式类似于下面这种。

HLS协议的例程如下：

  1. 播放器请求一个m3u8索引文件，然后服务器返回一个m3u8文件，里面就包含了连续的ts文件的名称。
  2. 播放器根据索引文件的内容，请求里面的第一个ts文件，第二个ts文件，第三个ts文件。
  3. 等播放器收到几个连续的ts后（一般会缓存多个，防止网络抖动发生卡顿），就开始播放。
  4. 播放器请求几个ts后，再请求新的m3u8文件。
  5. 这是更新了的新的m3u8内的ts序号又会增加，播放器便可以去请求新的ts文件了。

播放器定期获取m3u8文件，就能获得新的ts列表，一个个连续的ts文件就源源不断的从服务器传到了播放器，使得画面能够连续流畅。

优点：

  1. HTML5可以直接播放，不需要独立的APP。
  2. 基于HTTP，无需考虑防火墙或者代理问题。
  3. 通过动态修改m3u8内容，可以实现平滑切换码率。

缺点：

  1. 由于先要缓存数个ts文件才能播放，所以可能造成延迟过高。
  2. 有可能使用短连接，每请求一个ts文件都要新建一条连接，造成额外开销。
