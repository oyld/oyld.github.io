HTTPS简介

本文是我在学习HTTPS的过程中记录的一些笔记，主要介绍HTTPS的一些核心概念。

认证

服务器需要通过某种方式来了解用户身份。一旦服务器知道了用户身份，就可以判定用户可以访问的事务和资源了。认证就意味着要证明你是谁。通常是通过提供用户名和密码来进行认证的。

基本认证

基本认证简单便捷，但并不安全。只能用它来防止非恶意用户无意间进行的访问。

问题：基本认证的缺陷是什么？

摘要认证

摘要认证遵循的箴言是“绝不通过网络发送密码”。客户端不会发送密码，而是会发送一个“指纹”或密码的“摘要”，这是密码的不可逆扰码。

问题：摘要认证的缺陷是什么？

认证技术是一种轻量级的安全保护，在需要安全级别更加高的应用场景中，网络购物，银行业务等，则需要依赖更加安全的技术，如HTTPS。

HTTPS

HTTPS几个关键的问题：解决了哪些问题？分别使用什么技术实现的？弄清HTTPS的本质很重要。

HTTPS能够提供下列功能：

  • 服务器认证(客户端知道它们是在与真正的而不是伪造的服务器通话)。
  • 客户端认证(服务器知道它们是在与真正的而不是伪造的客户端通话)。
  • 完整性(客户端和服务器的数据不会被修改)。
  • 加密(客户端和服务器的对话是私密的，无需担心被窃听)。
  • 效率(一个运行的足够快的算法，以便低端的客户端和服务器使用)。
  • 普适性(基本上所有的客户端和服务器都支持这些协议)。
  • 管理的可扩展性(在任何地方的任何人都可以立即进行安全通信)。
  • 适应性(能够支持当前最知名的安全方法)。
  • 在社会上的可行性(满足社会的政治文化需要)。

数字加密——对报文内容进行编解码

密码：对文本进行编码，使偷窥者无法识别的算法。一套编码方案——一种特殊的报文编码方式和一种稍后使用的相应解码方式的结合体。
密钥：改变密码行为的数字化参数。密码密钥会让一个密码机看起来好像是多个虚拟密码机一样，每个密码机都有不同的密钥值，因此其行为都会有所不同。

对称密钥加密系统：编 / 解码使用相同密钥的算法。代表算法：DES。

问题：对称加密的缺陷是什么？

不对称密钥加密系统：编 / 解码使用不同密钥的算法。代表算法：RSA。

问题：非对称加密的缺陷是什么？

数字签名——说明是谁编写的报文，同时证明报文未被篡改过

如果拆包后的摘要与节点 B 自己的摘要版本不匹配，要么就是报文在传输过程中被篡改了，要么就是发送端没有节点 A 的私有密钥(也就是说它不是节点 A)。

数字证书——证明服务器没有被伪造

在数字签名中，如果客户端中的服务器公钥被人修改（劫持者拿不到服务器私钥，但可以修改客户端公钥），并且客户端到服务器之间有一个劫持者，那么劫持者就可以假装成原始服务器和客户端进行通信，即冒充风险（劫持者私自更换了密钥和私钥）。为了防止这种情况的发生，服务器可以找证书中心（CA）对服务器的公钥进行认证。客户端请求时，服务器会返回经过认证的证书，客户端用CA的公钥进行解密，拿到服务器的公钥，接下来就可以和服务器安全通信了。
通过 HTTPS 建立了一个安全 Web 事务之后，现代的浏览器都会自动获取所连接服务器的数字证书。如果服务器没有证书，安全连接就会失败。
由于任何人都能随意生成公钥、私钥对，可能有人会利用这个来进行非法操作。因此数字证书的根本目的就是证明公钥、私钥对的合法性。

HTTPS细节

不使用SSL/TLS的HTTP通信，就是不加密的通信。所有信息明文传播，带来了三大风险。

  • 窃听风险（eavesdropping）：第三方可以获知通信内容。
  • 篡改风险（tampering）：第三方可以修改通信内容。
  • 冒充风险（pretending）：第三方可以冒充他人身份参与通信。

SSL/TLS协议是为了解决这三大风险而设计的，希望达到：

  • 所有信息都是加密传播，第三方无法窃听——对应数字加密技术。
  • 具有校验机制，一旦被篡改，通信双方会立刻发现——对应数字签名技术。
  • 配备身份证书，防止身份被冒充——对应数字证书技术。

其中，安全层的作用是加密HTTP层的数据和解密TCP层的数据。在TCP连接建立之后，需要在安全层进行握手，对加密参数进行沟通，对各自的SSL/TLS层进行初始化。

整个HTTPS的流程是这样子的：

  • 客户端向服务器端索要并验证公钥。
  • 双方协商生成”对话密钥”。
  • 双方采用”对话密钥”进行加密通信。

步骤1和步骤2就是SSL/TLS握手过程了。实际上，在SSL/TLS握手完毕之后，客户端和服务端之间会采用对称加密技术对传输的数据进行加解密，而采用的密钥就是对话密钥。因此，对话密钥的保密工作很重要，不能被他人窃取，因此，对话密钥的生成过程就需要依赖非对称加密技术，所以在步骤1中，客户端向服务端索要的公钥，就是为了保护对话秘钥而存在的。这便是混合加密技术的应用。

混合加密系统和会话密钥：任何人只要知道了其公开密钥，就可以向一台公共服务器发送安全报文，所以非对称的公开密钥加密系统是很好用的。两个节点无须为了进行安全的通信而先交换私有密钥。 但公开密钥加密算法的计算可能会很慢。实际上它混合使用了对称和非对称策略。 比如，比较常见的做法是在两节点间通过便捷的公开密钥加密技术建立起安全通信， 然后再用那条安全的通道产生并发送临时的随机对称密钥，通过更快的对称加密技术对其余的数据进行加密。

具体的SSL/TLS的握手过程是这样的。会话秘钥就是在第三步生成的。
